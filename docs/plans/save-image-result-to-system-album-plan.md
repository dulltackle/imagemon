# 保存图片结果到系统相册实现计划

本文描述手机端把单张图片结果保存到系统相册的实现计划。目标是让设备使用者能从单张图片结果详情页或任务历史详情中的单张关联图片入口，显式把图片文件复制到系统相册，形成应用外非托管副本。

## 已确认范围

- “保存到系统相册”属于图片结果导出，只复制单张图片结果文件本身。
- 导出副本不包含任务快照、完整提示词、模型配置或任何应用内长期状态。
- 应用不管理系统相册里的外部副本，也不记录“已保存”状态。
- 重复保存允许产生多个外部副本；仅在当前保存进行中临时禁用按钮，避免并发触发。
- 保存时不压缩、不转码、不生成设备使用者可编辑文件名，交由系统相册保存原文件。
- 保存成功只显示当前页面一次性反馈，不跳转、不打开系统相册。
- 权限拒绝、写入失败只显示当前页面一次性失败信息，不写入任务历史、不改变图片结果状态、不产生长期提示。
- 不支持系统相册写入的平台保留按钮但禁用，提示当前平台不支持。
- 图片结果原图文件缺失或不可读取时保留按钮但禁用，提示图片文件缺失，无法保存到系统相册。

## 明确不做

- 不做自动保存。
- 不做批量保存或“一键保存本次任务全部图片”。
- 不创建 Imagemon 专属相册。
- 不实现分享或导出到文件系统。
- 不在 Web 上把“保存到系统相册”模拟为下载文件。
- 不从系统相册读取或浏览已有照片。
- 不保存系统相册 asset id、相册路径或外部副本引用。

## 依赖决策

- `CONTEXT.md`：图片结果、图片结果导出、任务历史。
- ADR 0042：图片结果可导出为应用外副本。
- ADR 0065：权限在设备使用者执行相关动作时按需请求。
- ADR 0117：图片结果原图文件缺失时禁用文件依赖操作。
- ADR 0182：图片结果导出只支持单张粒度，入口出现在单张图片结果详情和任务历史详情中每张关联图片上。

## 外部参考

- Expo MediaLibrary 官方文档：<https://docs.expo.dev/versions/v54.0.0/sdk/media-library/>

实现时优先使用 Expo SDK 匹配版本安装：

```bash
npx expo install expo-media-library
```

## 交互状态

每个保存入口至少需要表达以下状态：

- `ready`：文件可读取、平台支持，可以点击保存。
- `saving`：本次保存进行中，按钮临时禁用并显示保存中。
- `unsupported`：当前平台不支持保存到系统相册，按钮禁用。
- `missingFile`：图片结果原图文件缺失或不可读取，按钮禁用。

一次性反馈：

- 成功：`已保存到系统相册。`
- 权限拒绝：`未获得相册写入权限，无法保存。`
- 写入失败：`保存到系统相册失败，请稍后重试。`
- 平台不支持：`当前平台不支持保存到系统相册。`
- 文件缺失：`图片文件缺失，无法保存到系统相册。`

## 数据流

1. 设备使用者在图片详情页点击“保存到系统相册”，或在任务历史详情的某张关联图片项点击同名入口。
2. 页面先确认当前图片结果文件 URI 可解析。
3. 如果平台不支持，入口保持禁用并展示不支持原因。
4. 如果文件缺失，入口保持禁用并展示缺失原因。
5. 点击后进入 `saving`，同一入口在保存完成前不可再次触发。
6. 应用按需请求系统相册写入权限，只请求写入能力。
7. 权限通过后，把当前图片结果原文件 URI 交给系统相册保存。
8. 成功或失败只更新当前页面的一次性反馈；不写入任何持久仓储。
9. 保存结束后退出 `saving`。后续再次点击允许创建新的外部副本。

## 可执行子任务与提交边界

### 1. 记录术语与实现计划

Commit 标注：`docs: 记录图片结果保存到系统相册计划`

改动范围：

- 在 `CONTEXT.md` 中补充 `图片结果导出` 术语。
- 新增本文档，记录已确认范围、失败语义、入口范围和提交拆分。

验收：

- 文档清楚说明保存到系统相册是图片结果导出的一种形式。
- 每个后续子任务都有独立 commit 标注。

### 2. 接入 Expo MediaLibrary 配置

Commit 标注：`build: 接入系统相册写入依赖`

改动范围：

- 在 `apps/mobile/package.json` 增加 `expo-media-library`。
- 更新根 `package-lock.json`。
- 在 `apps/mobile/app.json` 配置 `expo-media-library` plugin。
- 配置中文保存权限说明，表达“用于把图片结果保存到系统相册”。

验收：

- `npm run mobile:typecheck` 能解析新增依赖类型。
- iOS 配置包含保存照片权限文案。
- 不新增读取相册、相机或位置相关能力。

### 3. 封装图片结果相册保存适配器

Commit 标注：`feat: 封装图片结果保存到系统相册能力`

改动范围：

- 新增或扩展 `apps/mobile/src/image-tasks` 下的导出相关模块。
- 定义面向页面使用的接口，例如 `ImageResultAlbumSaver`。
- Expo 实现负责：
  - 判断当前平台是否支持系统相册写入。
  - 按需请求写入权限。
  - 调用 MediaLibrary 保存本地文件 URI。
  - 将权限拒绝、平台不支持、文件缺失和写入失败归一为可展示错误。
- 内存实现用于测试和 Web/非原生兜底。
- 在 `AppRuntimeProvider` 中注入该适配器。

验收：

- 单元测试覆盖权限拒绝、平台不支持、保存成功和保存失败。
- 适配器不读取或返回系统相册资产列表。
- 适配器不写入图片结果仓储或任务历史仓储。

### 4. 在图片详情页加入保存入口

Commit 标注：`feat: 图片详情支持保存到系统相册`

改动范围：

- 在 `apps/mobile/app/images/[id].tsx` 加入“保存到系统相册”按钮。
- 使用当前图片结果解析出的 `imageUri` 作为保存源。
- 文件不可用时按钮禁用并显示缺失提示。
- 平台不支持时按钮禁用并显示不支持提示。
- 保存进行中临时禁用按钮，避免并发触发。
- 成功、权限拒绝和失败只在当前页面显示一次性反馈。

验收：

- 图片文件可用时能触发保存流程。
- 图片文件缺失时不能触发保存流程。
- 连续快速点击不会并发执行多次保存。
- 离开页面后不保留成功或失败反馈。

### 5. 在任务历史详情加入单张保存入口

Commit 标注：`feat: 任务历史支持保存单张关联图片到相册`

改动范围：

- 在 `apps/mobile/app/history/[id].tsx` 的每个关联图片结果项上加入保存入口。
- 每张图片结果独立解析文件 URI 和保存状态。
- 每张图片结果独立防抖，保存进行中只禁用对应入口。
- 沿用图片详情页同一适配器和同一反馈语义。

验收：

- 任务历史中多张关联图片可以分别保存。
- 单张保存进行中不阻塞其他图片项的页面浏览。
- 文件缺失、平台不支持、权限拒绝和写入失败均按单张结果展示。

### 6. 补齐测试与验证

Commit 标注：`test: 覆盖图片结果保存到系统相册流程`

改动范围：

- 为保存适配器补充单元测试。
- 为图片详情页和任务历史详情页可测试逻辑补充测试，优先覆盖状态计算与防抖逻辑。
- 必要时抽出纯函数，避免为了测试改动页面断言或绕过失败。

验收：

- `npm run mobile:test` 通过。
- `npm run mobile:typecheck` 通过。
- 不修改既有测试断言、Mock、Fixture 来规避失败。

### 7. 手动真机或模拟器验收

Commit 标注：`chore: 验证系统相册保存流程`

改动范围：

- 仅提交必要的验证记录或文档更新；如果无需落库文件，则不强行提交。

验收清单：

- iOS：首次保存触发写入权限请求，允许后保存成功。
- iOS：拒绝权限后显示当前页面失败信息，可重试。
- Android：保存成功后图片出现在系统相册或默认图片目录。
- Web：按钮存在但不可执行，显示当前平台不支持。
- 图片文件缺失：按钮存在但不可执行，显示文件缺失。
- 连续快速点击：只触发一次进行中的保存。

## 推荐实施顺序

1. 先提交文档与术语，冻结共享理解。
2. 再提交依赖和原生配置，单独处理 package 与 app config 变更。
3. 然后提交保存适配器，先把平台能力和错误语义封装稳。
4. 再分别提交图片详情页和任务历史详情页入口，避免 UI 改动过大。
5. 最后提交测试补齐和手动验收记录。

## 风险与注意事项

- Expo MediaLibrary 的原生权限配置需要重新构建原生应用后生效，不能只依赖 JS reload。
- 保存到系统相册只应使用本地文件 URI；内部文件路径必须先通过现有图片结果文件存储解析。
- 不要把系统相册 asset id 作为应用内状态保存，否则会改变“应用外副本不受应用管理”的边界。
- 不要在任务历史详情中做批量编排；每个入口只处理一张图片结果。
- Web 不要降级成文件下载，以免混淆“保存到系统相册”和“导出到文件系统”。
