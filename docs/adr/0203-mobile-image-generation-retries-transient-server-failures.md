# 移动端图片生成请求对瞬时服务端失败自动重试

移动端图片模型客户端（`apps/mobile/src/image-tasks/model-client.ts`）在图片生成 `POST /images/generations` 请求收到瞬时服务端失败时自动重试，最多 3 次尝试（首发 + 2 次重试），指数退避 500ms、1000ms。瞬时失败仅指 HTTP 408、429 与全部 5xx；重试用尽后按既有映射保存失败摘要（含状态码与平台码）。以下情形一律不重试：4xx 拒绝（服务端已明确判定请求无效或鉴权失败）、2xx 但响应体无效（生成可能已计费，重试会重复扣费）、图片 URL 下载失败、以及网络层抛错（移动端网络异常多意味着连接状态变化，立即反馈优于盲试，且请求可能已在服务端产生副作用）。调用方设置的应用侧超时信号横跨全部尝试；退避等待结束后若信号已中止，按 `timeout` 结束，不再发起下一次请求。

背景：实测 OpenAI 兼容中转网关（如 tu-zi）在上游图片模型饱和或超时时，会在长时间等待后返回 5xx，同一请求稍后重发即可成功。根包 CLI 走 Node OpenAI SDK，SDK 默认对连接错误与 408/429/5xx 自动重试 2 次，因此 CLI 侧极少暴露这类失败；移动端裸 `fetch` 适配层（依 0218 与竖切计划不复用 SDK）此前没有任何重试，单次瞬时 5xx 就直接以「模型服务暂时不可用」终结任务，形成 CLI 稳定、移动端易失败的体验落差。本决策让移动端与 CLI 的传输层韧性对齐。

本决策与任务级「不提供重试或再次执行」（生成任务竖切计划）不冲突：那是历史任务的 UI 资产管理动作，本决策是单次模型调用内部的传输层重试，任务历史仍只产生一条记录，模型调用锁在整个重试期间持续持有，符合 0218 前台单调用模型。

权衡：重试将最坏情况下的失败反馈时间延长约 1.5 秒加两次请求耗时；429 场景下的短退避未必足以越过限流窗口，但请求本身廉价且与 SDK 行为一致。重试次数与退避参数为实现常量，暂不开放配置；若后续需要按平台调优，另行记录决策。
