# 表格备份目标使用绑定标识并要求显式认领旧表

表格备份的写入语义是全量镜像：远端存在而本机不存在的记录会被删除。因此，表名和字段契约只能说明一张数据表“看起来可兼容”，不能证明它就是当前设备原先绑定的写入目标。我们把备份目标身份定义为 `(appToken, tableId, backupBindingId)`：已保存的 `tableId` 是正常路径的强身份；`backupBindingId` 是应用生成的非秘密 UUID，只在 `tableId` 丢失、明确失效或建表结果未知时用于重新发现同一目标；个人授权码只是访问凭据，不属于资源身份。同一 Base 轮换个人授权码不得改变目标身份，切换 Base 或显式新建目标才轮换 binding 并清除旧目标的成功时间。

新建数据表时，应用把 `__imagemon_backup_target_v1__<backupBindingId>` 文本字段与业务字段放在同一次建表请求中原子创建。解析器识别通用版本前缀：唯一且与本地 binding 匹配的 v1 marker 可以自动恢复绑定；多个匹配 marker、单表多个 marker 或未知更高版本都停止写入；表被重命名后仍可按 marker 找回。marker 不进入记录映射、镜像 diff 或恢复载荷。选择字段 marker 而不是哨兵记录，是因为后者需要建表后的第二次写入，无法覆盖“建表成功、写哨兵前中断”的窗口，也可能被全量镜像删除。

没有 marker 的旧 7/8/9/10 字段表、带其他 binding 的 Imagemon 表，以及使用者通过精确 `table` 参数指定的兼容表，都只能作为只读候选。应用不得按名称取第一张表、不得在确权前补字段或 marker、不得上传附件或增删改记录。使用者必须明确选择“先从此表恢复”“使用此表并以本机内容覆盖”或“保留此表并创建新目标”；覆盖还需二次破坏性确认并重新校验候选。恢复方向的显式选择只保存读取目标，不修改远端；备份方向的显式认领才允许补 marker 与缺失字段。

建表前必须先持久化 binding 与待建表名。收到 `TableNameDuplicated`，或建表发生超时、网络错误、5xx、无效响应、缺少 `table_id`、请求发出后取消时，本次不盲目重发非幂等 POST，而是只读扫描并按 marker 对账；无法唯一确认时保留 pending 状态并停止写入。正常备份仍直接使用已保存的 `tableId`，只有明确的 `TableIdNotFound` 才进入失效发现流程，网络、权限、服务端错误或数据未就绪都不得清除绑定或创建新表。

后果是首次连接旧 Base 或跨设备恢复会增加一次只读候选选择，且管理字段对使用者可见；换来的安全边界是：身份不确定时零记录写入，建表结果未知可以收敛到原表，`TableNameDuplicated` 不再形成永久循环，新设备也可以先恢复灾备数据而不被引导先做可能清空远端的镜像。
