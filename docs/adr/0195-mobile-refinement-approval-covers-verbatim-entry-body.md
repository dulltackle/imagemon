# 手机端提炼批准必须覆盖将写入的逐字正文

手机端模板提炼确认提炼方案（0022）时，必须同时展示将要写入个人图鉴条目的**完整正文逐字内容**和输入声明，并将其纳入使用者批准范围；使用者批准的不仅是精简方案的语义与范围，还包括落盘的逐字执行语义。确认后任何对正文或输入声明的调整都使确认失效，必须重新展示更新后的逐字正文并重新等待一次有效确认；应用不在确认后自行或借模型「调整措辞」而不再复核。

本决策是手机端对 Builder `references/proposal-format.md` 的**有意偏离**，原因是信任边界不同：

- Builder 由受 `SKILL.md` 约束的 Agent 执行，proposal-format 允许只展示精简方案、「用户确认的是方案语义和范围，不是逐字措辞」，并允许「确认后为清晰度、可执行性和去重调整措辞」——因为可信 Agent 被信任去写一份忠于方案的正文，并自行运行 `refinement-policy.md` 的语义校验（删除原任务具体内容、危险指令、URL、模型专属参数）。
- 手机端按 0019 把提炼交给**默认文本模型配置**，即任意 OpenAI 兼容、用户自配的文本模型（0032），它不保证遵守 refinement-policy；而应用只能执行 Promptdex 模板契约（`template-contract.md`）的**结构**校验（frontmatter 形状、正文非空、只引用已声明输入），**无法**语义校验「原任务具体内容是否删净」「是否残留危险指令、URL 或来自不可信外部提示词的注入」。外部完整提示词是不可信素材（0002），经任意模型处理后，注入或残留内容可能进入正文却照不进精简摘要。

因此手机端把语义保真度的最终 backstop 显式落在**使用者对逐字正文的批准**上：写什么就批准什么。这样即便文本模型薄弱、被注入或产出与摘要不符的正文，落盘成永久不可变执行语义（0053）前，使用者已逐字过目。精简方案仍按 proposal-format 展示用途、任务类型、最小输入集合、保留/删除摘要和提炼补充（0022），作为帮助使用者理解正文的导览，但不替代对逐字正文的批准。

本决策不改变其余提炼规则：提炼补充仍需明确标注并获批（0022），写入前仍做契约结构校验，名称冲突仍按 0185 写入前改名并重新确认，写入后清除提炼草稿且不保留外部完整提示词来源（0075），个人图鉴条目写入后执行语义不可变（0053）。重新生成方案仍是模型调用（0184），本地确认写入不是模型调用。

后续若共享核心（0009）引入可信的应用侧语义校验（例如可靠的残留具体内容或注入检测），可重新评估是否放宽逐字正文批准，并另行记录决策。
