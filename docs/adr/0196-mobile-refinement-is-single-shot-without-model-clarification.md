# 手机端首版提炼是单发，不引入模型主导的多轮澄清

手机端首版模板提炼是单发流程：使用者提供一个外部完整提示词和计划用途后，提炼模型一次性产出提炼方案（含将写入的逐字正文与输入声明，0195）。首版不引入 Builder `SKILL.md`/`refinement-policy.md` 中「信息不足时先追问」「无法可靠理解且可能影响图鉴条目身份的内容必须向用户澄清」那种由模型主导、向使用者提问的多轮澄清对话。

这是手机端对 Builder 提炼流程的有意收敛。Builder 由可信 Agent 执行，澄清是一轮自然的多轮交互，用于在产出方案前把图鉴条目身份定准（计划用途定义身份，优先级高于外部完整提示词）。手机端把提炼交给默认文本模型配置（0019），多轮澄清会带来明显复杂度：每轮提问都是一次模型调用、受全局模型调用锁约束（0029），提炼草稿（0074）还需持久化对话状态，与首版「保持等待态、错误展示和恢复语义清晰简单」（0029/0030）的基调冲突。

计划用途不足或方案身份跑偏时，使用者通过修改计划用途后**重新生成方案**（0184，模型调用）逐步逼近，而不是由模型发起澄清提问。含糊或可能残留的内容由两道既有门控兜底：使用者对逐字正文的批准（0195）和对全部提炼补充的批准（0022）；提炼模型仍可在单发方案中以提炼补充形式提出对身份的假设，但任何提炼补充都必须明确标注并获批后才写入。

权衡：单发在计划用途模糊时可能需要多次重新生成，每次都是一次模型调用；换取首版锁占用、提炼草稿状态和提炼页交互的简单稳定。后续若要引入模型主导的澄清轮次，需要重新评估多次模型调用的锁语义、草稿对话态持久化和提炼页交互，并另行记录决策。
