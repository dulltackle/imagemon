# 测试连接探测除非鉴权被拒或传输失败否则判通过

测试连接探测（0067/0193）的通过与失败只由两件事决定：服务器是否可达、凭据是否被显式拒绝。仅当探测返回显式鉴权拒绝（HTTP 401、403、429）或发生传输级失败（连接失败、DNS 失败、TLS 失败、超时、HTTP 5xx、响应无法作为 HTTP 响应解析）时，测试连接判失败并按 0138 保存对应非敏感失败摘要；探测返回 2xx，以及一切**非鉴权、非传输**的响应（包括探测端点返回 404、400 等，典型如平台根本没有 `/models` 端点）一律判通过，配置在字段完整且已保存 API Key 时标记为就绪。模型名与端点的真实有效性交首次真实图片任务或模板提炼调用裁决，失败按 0025 保存错误摘要。

本决策精确化 0193 中「凭据被接受」的判定，消除其对「没有 `/models` 的平台」这一核心目标场景的歧义。0193 让没有 `/models` 的兼容平台也能就绪，但没有 `/models` 时探测得到的是 404，而 404 给不出「凭据被接受」的正信号，无法区分「凭据有效、只是无此端点」与「base URL 错误」。若把 404 判失败，0193 想支持的无 `/models` 平台又被挡在就绪之外，0193 的意义落空；因此采用宽松判定：不要求探测正面确认凭据，只在凭据被**显式拒绝**或服务器**不可达**时才拦截就绪。这与 0193、0183、0188「应用只在确有权威信号时本地拦截，否则交服务端权威裁决」的一贯思路一致。

测试连接探测端点级 404 不按 0138 的 `not_found` 处理。0138 的错误类型枚举中 `not_found` 用于真实业务调用的服务端 404，以及（已被 0193 撤销的）模型名不在列表场景；测试连接探测自身返回 404（探测端点缺失或不被识别）不视为失败，不映射为 `not_found`，按本决策判通过。`unauthorized`（401）、`forbidden`（403）、`rate_limited`（429）、`server_error`（5xx）、`timeout`、`network_error`、`invalid_response` 等失败映射对测试连接探测继续适用。

权衡：宽松判定下，base URL 写错但服务器恰好返回非鉴权响应（如 404）的配置会被标为就绪，假阳性要到首次真实调用才暴露并保存错误摘要；换取对无 `/models` 或 `/models` 不完整的 OpenAI 兼容平台的支持，与 0032/0068 承诺一致。后续若为特定已知平台引入更强的正面凭据确认探测，需要另行记录决策。
