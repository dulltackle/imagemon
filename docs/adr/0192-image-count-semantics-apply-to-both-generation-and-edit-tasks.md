# 多图数量语义对生成任务与编辑任务一致适用

图片规格的数量是所有图片任务的统一交付属性（见 CONTEXT.md「图片规格」），不区分生成任务与编辑任务。0180（一条任务历史弱引用多张图片结果）和 0181（图片数量映射为一次原子模型调用）在原文中以「生成任务」举例描述多图语义，但该语义对编辑任务同样成立：编辑任务的数量 N>1 时，同样映射为对模型服务的一次原子请求（OpenAI 兼容的 `n` 参数），一次任务始终对应一次模型调用，没有「部分成功」，只在 0024 的进行中、完成、失败、状态未知之间取值，并按 0180 弱引用同一条任务历史、产出实际返回的 M 张图片结果（M 可小于请求的 N）。

本决策消除「数量」语义在两类任务上的不一致：CONTEXT.md 把数量定义为所有图片任务的属性，而 0180/0181 只举生成任务的例子，留下了编辑任务 N>1 是否允许、是否同样为原子 `n` 调用的空白。统一裁决为一致适用，避免后续在编辑流程里出现一对一假设或另一套多图语义。0182（图片结果导出只支持单张粒度）相应地对生成与编辑产出的每一张图片结果都成立。

`n` 上限不纳入已知模型能力兼容校验。0188 的图片规格能力兼容校验只覆盖应用自带能力知识的维度（尺寸、`input_fidelity`、透明背景等），不包含每个模型对 `n` 的上限。编辑端点（`/images/edits`）在部分兼容平台对 `n` 的支持可能不同于生成端点，但应用没有可靠的逐模型 `n` 上限知识；沿用 0183、0188「应用只在确有能力知识时本地拦截，否则交服务端权威裁决」的思路，N>1 的编辑若不被某服务支持，按失败任务处理并保存 0025 错误摘要，而不是本地拦截。与模型能力无关的应用级数量合法性（数量至少为 1）仍始终校验（0188）。

后续若要为编辑任务单独限定数量、或把逐模型 `n` 上限纳入已知模型能力校验，需要另行记录决策。
