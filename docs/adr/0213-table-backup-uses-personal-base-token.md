# 表格备份使用多维表格个人授权码而非开放平台应用

表格备份需要移动端在无后端的前提下读写设备使用者的飞书多维表格。我们放弃开放平台正式服务端 API 所要求的自建应用通道（建应用、开 `bitable:app` 权限、发布、加协作者、每两小时换发 `tenant_access_token`），改用多维表格「个人授权码」（Personal Base Token）：设备使用者在目标多维表格的 自定义插件 → 获取授权码 中生成 `pt-` 令牌，应用以 `Authorization: Bearer pt-xxx` 直连 `https://base-api.feishu.cn/open-apis/bitable/v1/apps/:app_token/...`，路径与标准 bitable v1 同形。选择理由：设置成本从跨两个后台的多步流程降到一分钟；令牌只授予单个多维表格的访问权，天然最小权限；长期静态令牌的凭据心智与既有模型配置的 API 凭据完全同构，沿用同一套安全存储与不入备份规则。2026-07-15 的 spike 已验证该通道对建表、建字段、批量增删改查记录、删表全部放行。

代价与已知坑：该通道面向多维表格自定义插件开放能力，官方文档薄于正式服务端 API，若未来被收紧则回退自建应用方案；知识库（wiki）内的多维表格链接携带的是 wiki 节点 token 而非 `app_token`，而解析 wiki 节点恰恰需要开放平台应用权限，因此飞书连接配置采用双轨录入——云空间 `/base/` 链接直接解析，检测到 `/wiki/` 链接时引导设备使用者用「开发工具」插件查出 `app_token` 后粘贴，输入框同时接受链接与裸 token。
