# 模型配置就绪只验证连接与凭据，不依赖模型列表

模型配置测试连接（0067）判定就绪时，只验证 base URL 可达、连接可建立、API Key 被服务端接受，**不**把「模型名是否出现在平台 `/models` 列表中」或「平台是否提供可解析的 `/models`」作为就绪门槛。测试连接成功（字段完整、已保存 API Key、轻量探测确认连接与凭据可用）即可把配置标记为就绪；模型名是否真实有效，交由首次真实图片任务或模板提炼调用裁决，按 0179/0188/0183「应用只在确有能力知识时本地拦截，否则交服务端权威裁决」的一贯思路处理。

本决策修订 0067 的两处就绪门槛，消除它与 0068 的正面矛盾：

- **撤销「模型名不在列表即失败」**。0067 原规定「轻量探测返回可解析模型列表但不包含当前模型名时按失败处理（`not_found`），不能标记就绪」。改为：模型名不在返回列表中**不**作为测试失败，**不**阻断就绪；应用最多在模型配置详情页给出一条**非阻断**的「疑似模型名拼写错误」提示，帮助使用者自查，但不写入模型配置测试失败摘要、不改变就绪或默认身份。0138 中「轻量探测返回可解析模型列表但不包含当前配置模型名时错误类型使用 `not_found`」一并撤销。
- **收敛「无法确认兼容性」**。0067 原规定「平台不支持轻量探测、或返回内容无法解析为模型列表时，报告无法确认兼容性、不标记就绪」。改为：只要探测确认了连接可达且凭据被接受（未返回 `unauthorized`/`forbidden` 等鉴权失败），即视为测试通过并可标记就绪，即使返回体无法解析为模型列表、或平台没有 `/models`；模型清单的可解析性不再是就绪前提。鉴权被拒（HTTP 401/403）仍按 0138 映射为 `unauthorized`/`forbidden` 失败，网络错误、超时、5xx 等仍按 0138 既有规则处理为失败。

由此 0068（模型名称自由输入、**不依赖兼容平台的模型列表接口**、拉取失败不影响保存）、0032（支持任意 OpenAI 兼容 base URL、预设非白名单、未知模型透传）与 0188（已知模型本地校验、未知模型透传）在就绪判定上恢复自洽：`/models` 只服务于 0068 的「拉取模型列表辅助选择」和本决策的非阻断拼写提示，不再是事实白名单，没有 `/models` 的兼容平台也能在测试连接验证凭据后正常运行任务。

0067 中与就绪门槛无关的机制保持不变：测试连接仍是显式动作、受全局模型调用锁约束（0029）、保存时不自动测试、失败可手动重试、跨页继续（0149）、可取消本地等待（0146）、30 秒应用侧超时（0151）、字段与凭据变化后需重测（0129/0130）、就绪配置重测失败清除就绪与默认身份（0131/0141）。原「无法确认兼容性」这一独立终态被本决策收敛后，0128/0131/0132/0138/0140/0141 中针对「无法确认兼容性导致未就绪 / 不保存失败摘要 / 清除最近测试成功时间」的分支一律不再触发（这些条文中相关句子成为不可达死文本，以本决策为准），但它们对真实测试失败（鉴权、网络、超时、5xx 等）清除就绪、默认身份、最近测试成功时间并保存失败摘要的规则继续适用。

权衡：解耦后模型名拼写错误不再在测试连接阶段被拦截，而是推迟到首次真实调用时按失败任务暴露并保存 0025 错误摘要；换取对「`/models` 不全或缺失的 OpenAI 兼容平台」的支持，与 0032/0068 的承诺一致。后续若要对特定已知平台恢复严格的模型名校验，需要另行记录决策。
