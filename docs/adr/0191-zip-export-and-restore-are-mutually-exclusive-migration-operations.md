# ZIP 导出与恢复是互斥的应用级独占迁移操作

ZIP 备份导出（0106 的导出进行中）和 ZIP 恢复（0083 的恢复进行中）是一对应用级独占迁移操作：同一时间最多只能有其中一个进行，二者互相排斥，各自也不能并发自身。导出进行中不能发起恢复或另一次导出，恢复进行中不能发起导出或另一次恢复。应用在确认导出或确认恢复进入对应进行中状态前，必须检查另一迁移操作未在进行；若已有迁移操作进行中，则禁用本次确认并说明原因。

该决策补齐 0083 与 0106 的禁止清单。0083 与 0106 各自只列举了图片任务、模板提炼、模型配置测试连接、拉取模型列表、图鉴编辑、单条导入导出、图片结果删除、任务历史删除、应用默认规格修改等会修改应用数据的操作，但都未把「另一迁移操作」列入禁止项。其中导出在语义上是只读操作——它按当前本地数据重新读取整库生成备份（0077/0106），并不修改应用数据；因此「禁止会修改应用数据的操作」这一框架恰好漏掉了「恢复进行中再发起导出」这个最危险的组合：导出会把正处于恢复事务中途（0082）的不一致数据读出并打包成自相矛盾的备份。对称地，导出进行中发起恢复会在导出读取整库的途中改库，造成竞态与损坏备份。本决策显式规定这两类迁移操作互斥且不可并发自身，消除该漏洞。

模型调用锁（0029）保持独立，不并入迁移互斥。0083 与 0106 已规定：模型调用进行中时，使用者仍可进入恢复预检页或导出确认页查看将迁移的内容，只是确认按钮随全局模型调用锁禁用，模型调用结束后再基于当前本地状态重新校验并恢复可确认状态。这套「可查看、暂不可确认」的语义要保留，因此迁移互斥只约束两类迁移操作之间，以及各自进入进行中后对模型调用的阻止（0083/0106 既有规则），不改写模型调用锁本身的占用与可视化（0173–0175）。

迁移互斥与只读恢复模式（0007）一致：本地数据迁移失败进入只读恢复模式时，应用允许导出完整备份用于抢救，但禁止创建或修改资产；ZIP 恢复是写入操作，在只读恢复模式下不可发起，导出仍可发起，但导出期间同样适用本决策的迁移互斥。

后续若要支持迁移操作排队、后台迁移或部分并发，需要重新评估事务边界、文件暂存提交（0082）、备份内容一致性（0077）与全局锁语义，并另行记录决策。
